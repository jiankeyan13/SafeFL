# Core 层设计思路订正 03

> 审查时间: 2026-02-23 23:26  
> 审查范围: `core/` 全部文件 (~20个源文件)  
> 审查视角: **框架设计哲学** — 高效框架的核心是 *约定大于配置*、*关注点分离*、*零成本抽象* 与 *可组合性*

---

## 一、全局性问题：Runner 是 God Object

**文件**: `core/runner.py` (421行)

### 症状

`FederatedRunner` 承担了太多不属于"主控循环"的职责：

| 职责 | 是否应属于 Runner |
|------|:-:|
| Round 循环调度 | ✅ |
| 客户端选择 (`_select_clients`) 含攻击者采样逻辑 | ❌ |
| 数据管道构建 (`_setup_data_pipeline`) | ❌ |
| 模型工厂构建 (`_setup_model`) | ⚠️ 勉强 |
| 构建 DataLoader (`_build_client_dataloader`) | ❌ |
| 全局评估 + 攻击组评估 (`_eval_attack_groups`) | ❌ |
| 本地评估采样 (`_run_local_evaluation`) | ❌ |
| Checkpoint 保存 (`_save_checkpoint`) | ❌ |
| Metric 构建 (`_build_metrics`) | ❌ |

### 哲学违背

- **单一职责原则 (SRP)**: Runner 应该只是一个 *编排器 (Orchestrator)*，即"胶水层"。它不应包含任何领域算法逻辑（如攻击者采样策略）或 I/O 操作（如 checkpoint）。
- **"框架 vs. 脚本"之别**: 高效框架将每个可变点封装为 *策略对象*，而非将逻辑堆砌在一个巨型类中。当前 Runner 更像一个 *实验脚本* 而非框架组件。

### 核心建议

Runner 应只保留 `_setup()` 和 `run()` 中的顶层编排逻辑，将以下职责外移：

1. **Evaluation** → 独立的 `Evaluator` 类（负责 clean eval、attack group eval、local eval）
2. **Client Selection** → 将 `_select_clients` 含攻击者采样的逻辑回归 `AttackManager` 或独立的 `Selector`
3. **Checkpoint** → `Logger` 或独立的 `CheckpointManager`
4. **DataLoader 构建** → 不应在 Runner 中通过"创建临时 Client 然后销毁"来构建 DataLoader

## 二、临时 Client 反模式 (anti-pattern)

**文件**: `core/runner.py` L165-183

```python
def _build_client_dataloader(self, client_id, task, dataset_store, mode, attack_profile=None):
    temp_client = self.client_class(client_id, self.device, self.model_fn)
    loader = temp_client.data_load(...)
    del temp_client  # 创建后立即销毁
    return loader
```

### 哲学违背

- **最小惊讶原则**: `Client` 在语义上代表"联邦学习参与者"，创建一个 Client 仅仅为了调用它的 `data_load` 方法然后立刻销毁，是对抽象的滥用。
- **不必要的开销**: 每次调用都会执行 `model_fn().to(device)` 实例化一个完整的模型，然后立即丢弃。
- **依赖倒置**: Runner 对 Client 的内部实现细节（`data_load` 方法）产生了不必要的耦合。

### 核心建议

数据加载逻辑应该是一个独立的函数或工具类，而非绑定在 Client 实例上。`data_load` 的核心逻辑是 `Subset(store, task.indices)` + `DataLoader(...)` + 攻击投毒，这与 Client 的模型、设备没有任何关系。

## 三、AttackStrategy 基类钩子名与实际使用不一致

**文件**: `core/attack/base_strategy.py` vs `core/client/base_client.py`

### 症状

| 基类定义的钩子名 | Client 中实际查询的钩子名 |
|---|---|
| `poison_dataset(dataset, mode)` | `poison_dataset(dataset, mode)` ✅ |
| `poison_train(model)` | ❌ 未使用。Client 查询的是 `compute_loss`, `on_after_backward`, `on_after_step` |
| `poison_update(update, initial_weights)` | `modify_weights(final_state_dict, initial_weights)` ❌ 名字不匹配 |

### 哲学违背

- **接口契约 (Interface Contract)**: 基类定义的是框架与扩展者之间的契约。当实际运行时调用的方法名与基类定义的不一致时，基类就失去了"类型安全文档"的作用。
- **隐式 Duck Typing**: Client 通过 `hasattr(attack_profile, 'compute_loss')` 进行"能力检测"，绕过了基类的定义。这意味着任何新的攻击策略开发者如果只看 `AttackStrategy` 基类，**根本无法知道自己需要实现哪些方法**。

### 核心建议

统一钩子命名。`AttackStrategy` 基类应定义框架实际会调用的所有钩子：

```python
class AttackStrategy:
    def poison_dataset(self, dataset, mode) -> Dataset: ...
    def compute_loss(self, output, target, model, criterion) -> Tensor: ...
    def on_after_backward(self, model): ...
    def on_after_step(self, model): ...
    def modify_weights(self, final_weights, initial_weights) -> Dict: ...
```

Client 端改为直接调用，而非 `hasattr` 检测。

## 四、Registry 重复定义 Bug

**文件**: `core/utils/registry.py` L68, L74

```python
ATTACK_REGISTRY = Registry("Attacks")   # L68
ALGORITHM_REGISTRY = Registry("Algorithms")  # L70
METRIC_REGISTRY = Registry("Metrics")   # L72
ATTACK_REGISTRY = Registry("Attacks")   # L74  ← 重复！覆盖了 L68 的实例
```

### 分析

`ATTACK_REGISTRY` 被定义了两次。如果在 L68 和 L74 之间有任何模块在导入时向第一个 `ATTACK_REGISTRY` 注册了组件，这些注册会在 L74 被新实例覆盖后全部丢失。

当前之所以没有报错，是因为 Python 的模块导入顺序恰好使得所有 `@ATTACK_REGISTRY.register(...)` 装饰器都在 L74 执行之后才被执行。但这是一个**定时炸弹**。

## 五、Server.broadcast 的浅拷贝陷阱

**文件**: `core/server/base_server.py` L79-88

```python
def broadcast(self, selected_clients):
    global_weights = {k: v.cpu().clone() for k, v in self.global_model.state_dict().items()}
    package = {client_id: global_weights for client_id in selected_clients}
    return package
```

### 分析

所有 Client 收到的是**同一个 dict 对象的引用**。虽然 Client 在 `receive_model` 中调用了 `model.load_state_dict(state_dict, strict=True)` 不会修改传入的 dict，但这依赖于 PyTorch 的内部行为，不是一个安全的契约。

更重要的是，如果任何算法扩展（如个性化 FL）需要在 `broadcast` 之后修改 `package[cid]` 中的某个 tensor，所有 Client 的权重都会被意外修改。

### 哲学违背

**防御式编程**: 框架代码应该"对用户假设最少"。要么每个 Client 一份独立拷贝（安全但慢），要么明确标注为"只读共享"并用 `types.MappingProxyType` 或 frozen tensor 保护。

## 六、Trainer 硬编码问题

**文件**: `core/client/trainer.py`

### 症状

1. **损失函数硬编码**: `self.criterion = torch.nn.CrossEntropyLoss()`（L9）。框架不应假设所有任务都是分类任务。
2. **优化器硬编码**: `_get_optimizer` 返回 SGD，momentum=0.9, weight_decay=5e-4（L72-77）。这些应该是从 config 读取的。
3. **梯度裁剪硬编码**: `clip_grad_norm_(... max_norm=5.0)`（L39）。`max_norm` 应可配置。
4. **每 epoch 新建优化器**: `train_epoch` 每次调用都会 `_get_optimizer()`（L16），这意味着优化器状态（如 momentum buffer）在跨 epoch 时会丢失。

### 哲学违背

**约定大于配置 ≠ 硬编码**: "约定大于配置"意味着提供合理的默认值，但*始终允许覆盖*。当前的 Trainer 是纯硬编码，没有任何可配置点。

## 七、跨层边界的职责泄漏

### 7.1 算法工厂 (`algorithms/`) 构建 DataLoader

**文件**: `algorithms/fedavg.py` L27-32

```python
test_ds_store = dataset_store[server_test_task.dataset_tag]
server_dataset = Subset(test_ds_store, server_test_task.indices)
test_loader = DataLoader(server_dataset, batch_size=batch_size, ...)
```

算法工厂函数 (`build_fedavg_algorithm`) 直接操作数据层的 `DatasetStore` 和 `Subset`。这意味着**每个新增算法**都必须重复编写这段一模一样的 DataLoader 构建代码。

### 7.2 Runner 直接操作 data 层内部类型

**文件**: `core/runner.py` L25-36

Runner 直接导入 `data.constants`、`data.task_generator`、`data.datasets`，并在 `_setup_data_pipeline` 中直接实例化 `DirichletPartitioner`。这使得 core 层对 data 层的内部实现产生了硬依赖。

### 哲学违背

**依赖反转原则 (DIP)**: 高层模块 (core) 不应依赖低层模块 (data) 的具体实现。二者都应依赖抽象。当前 core → data 是直接的 import 依赖。

## 八、Evaluation 逻辑的碎片化

当前的评估逻辑分散在三个地方：

| 位置 | 评估什么 |
|------|---------|
| `BaseServer.eval()` | 全局模型在 server 测试集上的表现 |
| `BaseClient.evaluate()` | 单个 Client 的本地评估 |
| `FederatedRunner._eval_attack_groups()` | 攻击组在污染测试集上的 ASR |

### 哲学违背

- 评估逻辑分散在 Server、Client、Runner 三处，没有统一的 `Evaluator` 抽象。
- `_eval_attack_groups` 中重建了完整的攻击策略实例（L296-299），这说明 Runner 承担了本不属于它的攻击评估编排职责。
- `BaseServer.eval()` 和 `BaseClient.evaluate()` 中存在几乎一模一样的 inference 循环（收集 preds/targets → 计算 metrics），这是明显的 **代码重复 (DRY 违背)**。

## 九、context 字典的类型安全缺失

**文件**: `core/server/base_server.py` L95, `base_screener.py`, `base_aggregator.py`, `base_updater.py`

Server 防御流水线通过一个 `context: Dict[str, Any]` 在 Screener → Aggregator → Updater 三个阶段之间传递信息。

### 哲学违背

- **隐式契约**: context 中放什么 key、什么 value，没有任何类型约束或文档。FLAME 的 `clip_value` 和 `noise` 是通过 context 传递的，但只有看 FLAME 的实现才能知道这一点。
- **脆弱耦合**: 如果 Screener 产生的 key 名被修改，Updater 会静默地读到 `None`，不会有任何编译时或运行时提示。

### 核心建议

定义一个 `PipelineContext` dataclass，明确声明所有可传递的字段：

```python
@dataclass
class PipelineContext:
    clip_value: Optional[float] = None
    noise_sigma: Optional[float] = None
    rejected_indices: Optional[List[int]] = None
    # ...
```

## 十、utils 目录的职责混乱

`core/utils/` 中包含了以下文件：

| 文件 | 实际职责 | 是否属于 utils |
|------|---------|:-:|
| `logger.py` | 日志管理 | ✅ |
| `registry.py` | 组件注册表 | ✅ |
| `scheduler.py` | 学习率调度 | ✅ |
| `metrics.py` | 评估指标 | ⚠️ 应独立为 `core/metrics/` |
| `triggers.py` | 攻击触发器 | ❌ 应属于 `core/attack/` |
| `poison_dataset.py` | 投毒数据集 | ❌ 应属于 `core/attack/` |
| `mmap_dataset.py` | 内存映射数据集 | ❌ 应属于 `data/` |

### 哲学违背

**"utils" 是代码坏味道**: 当一个 `utils` 目录开始包含领域逻辑（如攻击触发器、投毒数据集），说明模块边界已经模糊。`triggers.py` 和 `poison_dataset.py` 本质上是攻击子系统的一部分，应与 `core/attack/` 放在一起。`mmap_dataset.py` 是数据层的优化组件，不应出现在 core 中。

## 总结：核心矛盾

当前 core 层的根本问题在于 **"框架" 和 "实验脚本" 两种身份的混淆**：

```
理想框架             当前实现
 ┌──────────┐        ┌──────────┐
 │ Runner   │        │ Runner   │  ← 同时是编排器、评估器、选择器、
 │(编排器)  │        │(God Obj) │     数据管道构建器、checkpoint 管理器
 ├──────────┤        ├──────────┤
 │策略对象们│        │硬编码逻辑│  ← Trainer 中的优化器/损失函数/裁剪
 │可插拔组合│        │无法替换  │
 ├──────────┤        ├──────────┤
 │清晰的    │        │隐式的    │  ← context dict、duck typing、
 │接口契约  │        │约定      │     基类与实际使用不一致
 └──────────┘        └──────────┘
```

**一句话**: 当前 core 层实现了正确的功能，但作为一个"框架"，它缺乏 *可预测性* (predictability)、*可组合性* (composability) 和 *可发现性* (discoverability)。
