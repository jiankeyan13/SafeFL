# SafeFL Core层架构设计反思与订正（第二版 · 深层问题）

> **撰写时间**: 2026-02-23  
> **审查对象**: `core/` 目录全量代码（含 `runner.py`, `client/`, `server/`, `attack/`, `utils/`）  
> **与上一版的关系**: 上一版订正（01）已识别出上帝对象、缺乏钩子、攻击入侵、资源管理四大宏观痛点。本版聚焦于**更深层的结构性设计谬误**——那些即便实施了前版建议，仍会在框架扩展时频繁绊倒开发者的系统性缺陷。

## 1. 抽象层次混乱：两套推理管道，逻辑重复

### 1.1 Server.eval 与 Client.evaluate 的隐性重复

**现象**：  
`BaseServer.eval()` (line 141-171) 和 `BaseClient.evaluate()` (line 183-216) 各自独立实现了一套"加载模型 → forward → 收集预测 → 计算指标"的推理管道。虽然二者服务于不同场景（全局评估 vs 局部评估），但**核心推理循环**（forward + 收集 preds/targets）是完全同构的。

**为什么这是设计层面的错误**：  
- **DRY 违背（Don't Repeat Yourself）**：两处各自写了一遍 `torch.no_grad()` + `for data, target in loader` + `cat` 逻辑。一旦将来需要支持多 GPU 推理、混合精度推理、或者流式推理（大模型），必须同步修改两处。  
- **职责不对称**：Server 直接操控模型做推理（server 既管聚合又管推理），而 Client 通过委托 `StandardTrainer.inference()` 做推理。同一框架里同一件事有两种实现路径，对贡献者而言是认知毒药。

**高效框架的做法**：  
抽取一个统一的 `Evaluator` 或将推理统一收编到 `StandardTrainer.inference()`。Server 和 Client 的 eval 都**只负责准备 DataLoader 和 Metric**，然后委托给同一个推理引擎。

---

### 1.2 BN 校准逻辑碎片化三处

**现象**：  
BN 校准（Batch Normalization Calibration）在代码中以极相似的形式出现了**三次**：
1. `BaseClient.calibrate_bn()` (line 89-101) — 客户端本地校准
2. `BaseUpdater.calibrate_bn()` (line 30-39) — 服务器全局更新后校准
3. `FederatedRunner._setup_server_proxy_loader()` 构造的 proxy_loader 间接触发 updater 的校准

三处的核心逻辑都是 `model.train() → no_grad → forward(data)`，但行为细节不同（Client 限 5 batches，Updater 不限）。

**后果**：  
研究者如果想修改校准策略（如使用 Exponential Moving Average 而非 BatchNorm running stats），需要找三个地方改，且极容易遗漏。

**高效框架的做法**：  
BN 校准应该是一个独立的 **utility function** 或 **Calibrator** 组件，接受 `(model, loader, max_batches)` 三个参数，在全框架中只实现一次。

## 2. 注册表系统（Registry）设计缺陷

### 2.1 `ATTACK_REGISTRY` 被声明了两次

**现象**：  
在 `registry.py` 中，`ATTACK_REGISTRY` 在 line 68 和 line 74 各定义了一次：
```python
# line 68
ATTACK_REGISTRY = Registry("Attacks")
# ...
# line 74
ATTACK_REGISTRY = Registry("Attacks")
```
这意味着 line 68 的注册表会被 line 74 覆盖——**任何在两行之间导入并注册到 line 68 实例的组件都会丢失注册**。虽然目前代码碰巧没有触发此 bug，但它是一颗定时炸弹。

### 2.2 Registry 缺乏模块自动发现

**现象**：  
`runner.py` 的顶部大量 bare `import` 只是为了触发装饰器注册：
```python
import core.server.aggregator   # 仅为了让 @AGGREGATOR_REGISTRY.register 生效
import core.server.screener
import core.server.updater
import core.attack
import algorithms
import data.datasets
```

**为什么这是不好的**：  
- **隐式依赖（Implicit Dependency）**：删除某行 import，框架直接运行时 KeyError，但不会有任何静态分析工具能检测到。  
- **扩展性差**：每增加一个新的 aggregator/screener/attack，开发者除了写代码和装饰器，还要记得到 runner.py 加 import，否则就注册不上。  

**高效框架的做法**：  
使用 **`importlib` 自动扫描** 或 **entry_points / 配置文件声明式注册**。例如 MMCV 的 `Registry` 支持 `init_default_scope()` 自动扫描指定 package 下的所有模块。

### 2.3 ALGORITHM_REGISTRY.build 的特殊化

**现象**：  
`ALGORITHM_REGISTRY.build()` 的返回值是 `(server, client_class)` 这一元组——与其他 Registry 返回单个实例的**统一接口**完全不一致。逻辑掩藏在 `algorithms/` 包里，`Registry.build()` 自己只是调用 `cls(**kwargs)`，这意味着 Algorithm 的构造函数返回值被要求为 tuple，而非 Algorithm 实例本身。

**为什么这是设计层面的错误**：  
- Registry 的 `build()` 承诺的语义是"构造并返回注册类的实例"。但 Algorithm 实际返回的是一个 tuple，破坏了接口一致性。  
- 未来若有人用 `ALGORITHM_REGISTRY.get("fedavg")` 获取类并期望它是一个正常类，行为就会出乎意料。

**高效框架的做法**：  
Algorithm 应被建模为一个 **Algorithm Builder / Factory**，它自身拥有 `.build_server()` 和 `.build_client_class()` 方法，而非在构造函数返回 tuple。

## 3. 数据流与控制流的纠缠

### 3.1 Runner 同时管理"数据管道"和"训练循环"

**现象**：  
`FederatedRunner` 既持有 `task_set`, `dataset_stores`（数据管道状态），又驱动 `run()`（控制循环）。这不仅仅是"上帝对象"问题（上版已提），更深层的问题是：**数据管道的生命周期被强绑定到了 Runner 的生命周期**。

**后果**：  
- 不可能复用同一份数据管道配置跑多次实验（如 Hyperparameter Sweep），每次都必须构建新的 Runner。  
- 不可能在 Runner 外独立测试数据管道（如单元测试 partitioner 是否正确划分）。  

**高效框架的做法**：  
数据管道应该是一个独立的 **`DataModule`**（参考 PyTorch Lightning 的 `LightningDataModule`），拥有自己的 `setup()` 和 `teardown()` 生命周期，与 Runner 完全解耦。

### 3.2 Client 被迫兼任"DataLoader 工厂"

**现象**：  
`runner.py` 的 `_build_client_dataloader()` (line 165-183) 创建一个**临时 Client**，只为了调用 `client.data_load()` 获取一个 DataLoader，然后立即 `del client`。

```python
temp_client = self.client_class(client_id, self.device, self.model_fn)
loader = temp_client.data_load(...)
del temp_client  # 创建一整个模型实例，只为了拿一个 DataLoader
```

**为什么这是荒谬的**：  
- `BaseClient.__init__` 会调用 `model_fn().to(device)`，构造一整个深度学习模型。这意味着**每次构建 DataLoader 都会产生一次完整的模型实例化开销**（GPU 显存分配 + 参数初始化）。  
- 这暴露了 `data_load` 本不应该是 Client 的方法——DataLoader 的构建不需要知道 Client 的存在。

**高效框架的做法**：  
`data_load` 应该是一个静态函数或 DataModule 的方法。Client 只需接收已经构建好的 DataLoader。

## 4. Trainer 抽象与配置管理的缺陷

### 4.1 Trainer 硬编码了唯一的优化器配置

**现象**：  
`StandardTrainer._get_optimizer()` (line 72-77) 硬编码了 SGD + momentum=0.9 + weight_decay=5e-4：
```python
return torch.optim.SGD(self.model.parameters(), 
                       lr=self.config.get('lr', 0.01), 
                       momentum=0.9, 
                       weight_decay=5e-4)
```
以及 `train_epoch()` 硬编码了 `clip_grad_norm_(max_norm=5.0)`。

**后果**：  
- 想换 Adam/AdamW？改源码。  
- 想调 momentum？改源码。  
- 想关闭 gradient clipping？改源码。  
- 每一次改动都违反 OCP（开闭原则）。

**高效框架的做法**：  
优化器、梯度裁剪等应全部 **config-driven**，从 `self.config` 读取 optimizer type、optimizer params、clip_norm，使得任何配置变更不需要触碰源代码。

### 4.2 Trainer 在每个 epoch 都重新构造 Optimizer

**现象**：  
`BaseClient.train()` 内循环中每个 epoch 都调用 `trainer.train_epoch()`，而 `train_epoch()` 内部每次都 `self._get_optimizer()` 重新构建优化器。

**后果**：  
- 优化器的动量（momentum buffer）在每个 epoch 之间丢失，等价于每个 epoch 都冷启动 SGD。  
- 如果使用 Adam，其一阶/二阶矩估计（m, v）在每个 epoch 重置，与"本地多 epoch 训练"的学术假设严重矛盾。  
- **这不仅是效率问题，更是正确性问题**——它静默地改变了本地训练的数学行为。

**高效框架的做法**：  
Optimizer 应该在 `Trainer.__init__` 或 `train()` 开始时构建一次，在所有 epoch 间复用。

## 5. 接口协议的不一致性

### 5.1 AttackStrategy 的方法命名与 Client 调用不匹配

**现象**：  
`AttackStrategy` 基类定义了三个钩子：
- `poison_dataset()` — Client 通过 `attack_profile.poison_dataset()` 调用 ✅  
- `poison_train()` — **Client 从未调用此方法**。Client 实际使用的是 `attack_profile.compute_loss` / `on_after_backward` / `on_after_step` 这三个属性名 ❌  
- `poison_update()` — 也未被调用，Client 实际调用的是 `attack_profile.modify_weights()` ❌  

**后果**：  
`AttackStrategy` 基类定义的接口协议是一个"谎言"——子类如果按基类规范实现 `poison_train()` 和 `poison_update()`，它们永远不会被框架调用。真正的协议散落在 `BaseClient` 的 `hasattr` 检查中，完全不可发现。

**高效框架的做法**：  
基类必须定义框架实际会调用的接口。如果框架通过 `hasattr` 鸭子类型检测方法存在性，至少在基类中提供默认空实现，并在文档中明确说明。

### 5.2 `hasattr` 导向型鸭子类型的脆弱性

**现象**：  
`BaseClient` 中到处都是 `hasattr(attack_profile, 'xxx')` 检查：
```python
if attack_profile is not None and hasattr(attack_profile, 'poison_dataset'):
if hasattr(attack_profile, 'compute_loss'):
if hasattr(attack_profile, 'on_after_backward'):
if hasattr(attack_profile, 'modify_weights'):
```

**为什么这比显式 `if/else` 更危险**：  
- 方法名的拼写错误不会产生任何编译期或运行期错误，它只是静默地跳过。  
- 无法使用任何 IDE 的"查找引用"或重构工具进行安全重命名。  
- 绝对没有类型安全。

**高效框架的做法**：  
定义明确的 `Protocol`（PEP 544）或 Abstract Base Class，方法以 `Optional` 的抽象方法存在，或使用显式的 `supports_xxx()` 标志。

## 6. 配置传递的反模式

### 6.1 config 字典的无限传播 (Config Soup)

**现象**：  
一个巨大的 `self.config` 字典从 Runner 出发，先 `copy.deepcopy` 打包为 `round_config`，然后流经 `client.execute() → client.data_load() → client.train() → StandardTrainer.__init__` 乃至更深层。每一层都用 `config.get('batch_size', 32)` 从同一个大字典里随意抽取自己需要的 key。

**为什么这是反模式**：  
- **无契约 (No Contract)**：没有任何地方声明 config 必须包含哪些 key。缺少一个 key 不会报错，而是静默使用默认值——这在研究实验中极其危险（"以为配了 lr=0.001 但其实 typo 成了 Lr=0.001，训练完才发现用的是默认 0.01"）。  
- **隐式耦合**：Client 和 Trainer "知道" config 的内部结构，但这种知识没有任何显式的文档或类型约束。  
- **deepcopy 性能开销**：每轮都 `copy.deepcopy(self.config['client'])`。如果 config 很大（比如包含嵌套的超参数搜索空间），这个开销是不必要的。

**高效框架的做法**：  
使用 **typed dataclass** 或 **Pydantic Model** 定义配置。例如：
```python
@dataclass
class ClientConfig:
    lr: float = 0.01
    batch_size: int = 32
    epochs: int = 1
    num_workers: int = 0
```
这样任何拼错字段名的行为都会在启动时直接报错，而非在训练 300 轮后才发现。

## 7. 内存模型与状态管理的系统性问题

### 7.1 broadcast 的共享引用陷阱

**现象**：  
`BaseServer.broadcast()` (line 79-88)：
```python
global_weights = {k: v.cpu().clone() for ...}
package = {client_id: global_weights for client_id in selected_clients}
```
所有 client 拿到的是**同一个字典对象的引用**（`global_weights` 只 clone 了一次）。

**后果**：  
在当前串行实现中碰巧安全（每个 Client 用 `model.load_state_dict(strict=True)` 加载后不再修改输入字典）。但如果未来任何扩展（如并行训练、或 Client 端在 `receive_model` 中修改字典），所有 Client 会互相踩踏同一份数据。这是一个**经典的浅拷贝陷阱 (Shallow Copy Trap)**，且代码注释中写了"deepcopy 防止意外修改"，但实际并未使用 deepcopy。

### 7.2 模型状态的隐式所有权不明

**现象**：  
`global_model` 被 Runner、Server 同时持有引用。Server 在 `step()` 中通过 `updater` 直接修改 `self.global_model` 的 state_dict。Runner 在 `_save_checkpoint()` 中直接读取 `self.global_model.state_dict()`。

**后果**：  
谁"拥有"模型不清楚。如果将来在 `step()` 和 `_save_checkpoint()` 之间插入异步操作（如后台日志写入），就可能遭遇竞态条件。

## 8. 总结：从第一版到第二版的递进

| 维度 | 第一版订正 (宏观) | 第二版订正 (深层) |
|------|------------------|------------------|
| 关注点 | 职责过大、缺钩子、攻击入侵、资源浪费 | 抽象重复、接口谎言、配置反模式、正确性 Bug |
| 严重等级 | 影响"可维护性与扩展性" | 影响"正确性与可靠性" |
| 典型代价 | 修改一个功能要改 N 个文件 | 实验结果无法复现、静默出错 |
| 解决方向 | 引入 Callback/Builder/Decorator | 统一接口协议、Typed Config、修复逻辑 Bug |

### 优先级排序（建议修复顺序）

1. 🔴 **P0 - 正确性** : Trainer 每 epoch 重建 Optimizer（§4.2），直接影响训练数学行为
2. 🔴 **P0 - 正确性** : AttackStrategy 基类接口与实际调用不匹配（§5.1），新攻击实现者必踩坑
3. 🔴 **P0 - 潜在 Bug** : `ATTACK_REGISTRY` 重复定义（§2.1），已注册组件可能丢失
4. 🟡 **P1 - 架构** : Config Soup 无类型约束（§6.1），研究可复现性受威胁
5. 🟡 **P1 - 架构** : 推理管道重复（§1.1）+ BN 校准碎片化（§1.2）
6. 🟡 **P1 - 架构** : Client 兼任 DataLoader 工厂（§3.2），无意义的性能损耗
7. 🟢 **P2 - 改进** : Registry 缺乏自动发现（§2.2）
8. 🟢 **P2 - 改进** : broadcast 浅拷贝（§7.1），当前安全但未来不安全
