> **日期**: 2026-02-23
> **范围**: `data/` 目录全部文件（7 个源文件 + 1 个子包）
> **评审标准**: Clean Architecture · Clean Code · SOLID

---

## 一、现有架构总览

```
data/
├── constants.py        # Split/Tag 字符串常量 + tag 工厂函数
├── registry.py         # 极简装饰器注册表（全局 dict）
├── dataset_store.py    # DatasetStore (Dataset 包装) + build_dataset 工厂
├── task.py             # Task 数据类 + TaskSet 容器
├── partitioner.py      # Partitioner 抽象基类 + IID / Dirichlet 实现
├── task_generator.py   # TaskGenerator 流水线控制器
└── datasets/
    ├── __init__.py     # 触发 cifar10 注册
    └── cifar10.py      # CIFAR10 构建函数 × 3
```

### 数据流向

```
cifar10.py 注册 ──► registry.py (dict)
                      │
runner.py ──► TaskGenerator.generate()
                 │
                 ├─ _load_sources() → build_dataset() → DatasetStore
                 ├─ _sample_proxy_indices()
                 ├─ _partition_remaining() → Partitioner.partition() → TaskSet
                 └─ _build_client_tasks / _build_server_tasks → 最终 TaskSet
                      │
              runner / client 消费 TaskSet + stores
```

## 二、优点（做得好的地方）

| 项目 | 说明 |
|------|------|
| **职责分离清晰** | `Task/TaskSet` 是纯数据结构、`Partitioner` 是策略接口、`TaskGenerator` 是流水线控制器，三者边界分明 |
| **开闭原则 (OCP)** | 新增数据集只需在 `datasets/` 下加文件 + `@register_dataset`，无需改动 `TaskGenerator` |
| **策略模式** | `Partitioner` 是标准策略模式，IID / Dirichlet 可互换 |
| **测试覆盖** | `tests/data/` 已覆盖 `TaskSet`、`Partitioner`、`TaskGenerator` 的核心路径 |
| **常量集中** | `constants.py` 消除了散落的魔法字符串 |

## 三、问题与改进方案

### 问题 1：`registry.py` — 全局可变 dict，无保护

**现状**：
```python
dataset_builders = {}  # 裸 dict，模块级全局变量
def register_dataset(name):
    def decorator(cls):
        dataset_builders[name] = cls
        return cls
    return decorator
```

**违反原则**：
- Clean Code §4 *Context Passing*：NEVER use global variables
- 无重复注册检测，静默覆盖
- 变量名 `dataset_builders` 中的 "builders" 不够精确（注册的是函数，不是 builder 类）

**改进方案**：

将裸 dict 封装为 `DatasetRegistry` 类，提供类型安全的注册与查找方法，并在重复注册时抛出异常。

```python
# registry.py
from typing import Callable, Dict
from data.dataset_store import DatasetStore

DatasetFactory = Callable[[str, bool], DatasetStore]

class DatasetRegistry:
    def __init__(self):
        self._factories: Dict[str, DatasetFactory] = {}

    def register(self, name: str):
        """装饰器：注册数据集工厂函数。重复注册=编程错误→直接抛异常。"""
        def decorator(func: DatasetFactory) -> DatasetFactory:
            if name in self._factories:
                raise ValueError(f"数据集 '{name}' 已注册，请勿重复注册")
            self._factories[name] = func
            return func
        return decorator

    def build(self, name: str, root: str, is_train: bool) -> DatasetStore:
        if name not in self._factories:
            raise KeyError(f"数据集 '{name}' 未注册")
        return self._factories[name](root, is_train)

# 模块单例
dataset_registry = DatasetRegistry()
```

**影响范围**：`cifar10.py`（装饰器改名）、`dataset_store.py`（`build_dataset` 改为调用 `dataset_registry.build`）。

### 问题 2：`dataset_store.py` — 职责混杂

**现状**：该文件同时包含：
1. `DatasetStore` 类（Dataset 包装器）
2. `_extract_targets_from_dataset` / `_extract_targets_from_subset`（标签提取工具函数）
3. `build_dataset` 全局工厂函数

**违反原则**：
- SRP：工厂函数与数据结构不应在同一文件
- Clean Code §2 *Extract Method*：`_extract_targets_*` 是 `DatasetStore.get_label()` 的内部实现细节，不应暴露为模块级函数

**改进方案**：

将 `build_dataset` 移入 `registry.py`（已重构为 `DatasetRegistry.build`），`_extract_targets_*` 改为 `DatasetStore` 的静态私有方法或保持为模块私有函数（当前已是 `_` 前缀，可接受）。

最终 `dataset_store.py` 只保留 `DatasetStore` 这一个类。

### 问题 3：`TaskGenerator` — 方法参数过多 + `_load_sources` 的异常处理

**现状**：

```python
# _load_sources 中 catch Exception 后 print + re-raise
except Exception as e:
    print(f"Error loading '{full_tag}': {e}")
    raise e
```

**违反原则**：
- Clean Code §5 *Exceptions > Error Codes*：`print` + `raise` 是噪音。调用者已有日志系统
- `generate()` 内部 4 处 `print()`，属于表现层逻辑混入 data 层

**改进方案**：

1. **移除所有 `print` 语句**：`TaskGenerator` 是纯 data 层，不应直接输出到 stdout。如需日志，由 runner 层注入 logger 或使用 Python `logging` 模块
2. **简化异常处理**：删除 `_load_sources` 中无用的 `except...raise`，让异常自然传播
3. **可选改进**：如果确实需要计时信息，可以返回一个 `GenerationResult` 数据类

```python
# 简化后的 _load_sources
def _load_sources(self) -> None:
    sources_config = [
        (train_aug_tag(self.dataset_name), True),
        (train_plain_tag(self.dataset_name), True),
        (test_plain_tag(self.dataset_name), False),
    ]
    for full_tag, is_train in sources_config:
        self.stores[full_tag] = dataset_registry.build(
            full_tag, self.root, is_train
        )
```

### 问题 4：`constants.py` — tag 工厂函数与常量混放

**现状**：`constants.py` 同时定义了：
- 字符串常量（`SPLIT_TRAIN`, `OWNER_SERVER`...）
- 工厂函数（`client_owner()`, `train_plain_tag()`, `test_plain_tag()`）

**违反原则**：
- SRP：常量文件不应包含逻辑函数
- 但函数极其简单（单行 f-string），过度拆分反而增加文件数

**评估**：**可暂不改动**。函数体只有 1 行，且与常量紧密相关。按 Clean Code §7 *Rule of Three*，目前无需提前抽象，维持现状是合理的。

---

### 问题 5：`TaskSet` — 使用硬编码字符串 `"server"`

**现状**：
```python
def list_client_ids(self, exclude_server: bool = True) -> List[str]:
    ids = [cid for cid in ids if cid != "server"]  # 硬编码
```

**违反原则**：
- 应使用 `constants.OWNER_SERVER`

**改进方案**：

```python
from data.constants import OWNER_SERVER

def list_client_ids(self, exclude_server: bool = True) -> List[str]:
    ids = list(self._tasks.keys())
    if exclude_server:
        ids = [cid for cid in ids if cid != OWNER_SERVER]
    return ids
```

### 问题 6：`cifar10.py` — `del is_train` 反模式

**现状**：
```python
@register_dataset("cifar10_train_aug")
def build_cifar10_aug(root: str, is_train: bool) -> DatasetStore:
    del is_train  # 固定为 train+aug
    return _build_cifar10_impl(root, is_train=True, use_aug=True)
```

**违反原则**：
- 签名承诺接受 `is_train` 参数，但内部立即丢弃，违反接口契约
- 这是 registry 接口设计的缺陷：`DatasetFactory = Callable[[str, bool], DatasetStore]` 强制所有工厂都接受 `is_train`，但部分数据集的变体已通过 tag 名称确定了 train/test

**改进方案（低优先级）**：

当前阶段可暂不改动——因为注册表的统一签名简化了 `build_dataset` 的调用。日后如果 `datasets/` 下数据集增多，可考虑将工厂签名改为仅接收配置 dict，或使用 `**kwargs`。

## 四、改动优先级汇总

| 优先级 | 问题 | 改动量 | 说明 |
|--------|------|--------|------|
| 🔴 高 | P3: `TaskGenerator` 移除 print 语句 | ≈10行删除 | 打破 data 层与 stdout 的耦合 |
| 🟡 中 | P1: `registry.py` 封装为类 | ≈25行重写 | 消除全局可变 dict |
| 🟡 中 | P2: `build_dataset` 移入 registry | ≈5行移动 | 单一职责 |
| 🟡 中 | P5: `TaskSet` 使用常量替换硬编码 | 1行 | 消除魔法字符串 |
| 🟢 低 | P4: `constants.py` 拆分 | 暂不改 | Rule of Three，暂无必要 |
| 🟢 低 | P6: `cifar10.py` 签名优化 | 暂不改 | 统一签名利大于弊 |

## 五、总体评价

> **当前 data 层架构整体是合理的。**

核心流水线（加载 → 分区 → 组装任务）的职责划分清晰：
- `Task`/`TaskSet` 是纯数据结构
- `Partitioner` 策略模式正确
- `TaskGenerator` 作为编排器的抽象层次恰当
- 注册表机制支持开闭原则

主要改进点集中在**代码卫生**层面（全局可变状态、print 噪音、硬编码字符串），而非架构层面的根本性问题。建议按上述优先级逐步清理，无需大规模重构。
