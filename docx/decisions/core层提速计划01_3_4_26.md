## core层提速计划01_3_4_26

### 背景
- **场景**: 单机多GPU仿真模拟, FedAvg(AvgAggregator) + ResNet18, 优先使用GPU加速.
- **结论**: 当前主要耗时不在算子本身, 而在 Python 串行调度, 大量 CPU<->GPU 同步拷贝, 聚合阶段内存峰值, 以及确定性训练开销.

### 目标
- **主目标**: 在不改变 FedAvg 数学含义的前提下, 显著降低单轮 wall time, 并能随着 GPU 数量增加而近似线性扩展.
- **约束**: 方案不依赖固定超参(客户端数, fraction, local_epochs 等可变).

### 现状瓶颈(按影响优先级)
- **B1, 客户端训练串行且反复构建对象**: 每轮对 selected clients 逐个创建 Client 并训练, 同时每次初始化都会创建 DataLoader, 导致 Python 开销大, 且多GPU无法并行吃满.
- **B2, 广播与 delta 计算存在全量 cpu().clone()**: 广播阶段将全局权重搬到 CPU 并 clone, 客户端为算 delta 在训练前后都进行全量 CPU 拷贝与 clone, 触发 GPU 同步与带宽浪费.
- **B3, 聚合阶段 torch.stack 产生巨大中间张量**: 对每层把所有客户端 delta stack 成 [num_clients, ...] 的大张量再加权求和, 内存峰值高且多次 dtype/device 搬运.
- **B4, 服务器更新路径额外拷贝**: 服务器侧构造 new_state(全量 clone) + load_state_dict, 每轮产生额外分配与拷贝.
- **B5, 每轮全局评估与可选 BN 校准**: 每轮全局 eval 会占用可观时间, 若启用 proxy BN 校准则额外进行一遍前向.
- **B6, 强制确定性算法**: deterministic + benchmark=False + 强制 deterministic_algorithms 会显著降低卷积网络吞吐.

### 提速方案(按优先级, 可分阶段落地)
#### P0, 立刻收益, 改动最小
- **S0, 关闭确定性慢开关**:
  - 允许非确定性路径, 开启 cudnn.benchmark.
  - 若可接受数值微小变化, 启用 TF32 以提升 matmul/conv 吞吐.
  - 预期收益: ResNet18 常见为明显加速, 且不会改变 FedAvg 的算法结构.

#### P1, 最大瓶颈修复, 重点降低 CPU<->GPU 同步与拷贝
- **S1, 广播不落 CPU**:
  - server broadcast 不再对每个参数做 cpu().clone().
  - 单机仿真中 server 与 client 同机同进程/同 worker 时, 可直接传递 GPU state_dict 或同设备 tensor 引用(注意只读语义).
- **S2, delta 在 GPU 上计算, 最后再按需转移或压缩**:
  - 不再保存 initial_state 的 CPU 副本.
  - 训练结束后用 current_params - received_global_params 在 GPU 上生成 delta, 必要时再转 CPU 或做压缩(如 fp16).
- **S3, 聚合改为流式累加, 避免 torch.stack 峰值**:
  - 对每层做 acc += w_i * delta_i, acc 用 fp32, 最终再 cast 回目标 dtype.
  - 尽量在 delta 已在的 device 上聚合, 降低 .to() 的搬运次数.

#### P2, 让多GPU真正并行, 获得可扩展加速
- **S4, 多进程并行仿真, 每GPU一个常驻 worker**:
  - 启动 N 个 worker, 每个 worker 固定 cuda:i.
  - 每轮 server 选人后将 client ids 分配到各 worker 并行训练, worker 内可顺序处理其分配的 clients.
  - worker 侧常驻复用 Client/DataLoader, 避免每轮反复构建.
  - server 收集各 worker updates 后做一次聚合与更新.
  - 预期收益: GPU 数增加时 wall time 显著下降, 且不需要为每个 client 单独进程.
- **S5, client 侧 AMP**:
  - 在本地训练使用 autocast + GradScaler(bf16 或 fp16), 提升吞吐并减少显存压力.

#### P3, 进一步优化输入与评估开销
- **S6, DataLoader 优化**:
  - 对 client 训练 DataLoader 启用 num_workers>0, pin_memory=True, persistent_workers=True, 合理 prefetch_factor.
  - 使用 non_blocking=True 的 H2D 拷贝与 pinned memory 配合.
- **S7, 评估降频或抽样**:
  - 将全局评估从每轮改为按 interval, 或对评估集做抽样以降低固定开销.
- **S8, BN 校准策略化**:
  - 仅在必要轮次或特定条件触发 BN 校准, 或缩小 proxy 规模.

### 推荐落地顺序(只做3件事时)
- **Step1**: S0 关闭确定性慢开关, 打开 benchmark/TF32.
- **Step2**: S1+S2 去掉 broadcast 与 delta 的 cpu().clone() 路径, 减少同步拷贝.
- **Step3**: S4 多GPU并行仿真(每GPU一个常驻 worker), 并配合 S3 流式聚合.

### 验收指标(建议记录到日志)
- **性能**: round time(p50,p90), samples/sec, GPU utilization, GPU mem peak, 聚合耗时, broadcast 耗时.
- **正确性**: 与当前实现相比, 指标趋势一致(允许非确定性时不要求 bit-level 一致), 且聚合结果满足 FedAvg 加权含义.

### 风险与注意事项
- **可复现性**: 关闭确定性会降低可复现性, 需用 seed + 统计意义上的重复实验替代 bit-level 对齐.
- **进程与显存管理**: 多进程 GPU worker 需避免重复占用同一 GPU, 并注意在 worker 内复用模型与 DataLoader.
- **张量所有权**: 共享/传递 GPU tensors 时要保证不会被下游 in-place 修改导致数据竞态, 建议明确拷贝策略或只读约束.
