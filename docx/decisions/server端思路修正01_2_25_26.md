# Server端设计批判与架构思路修正

> **评审说明**：本文档基于业界成熟的联邦学习框架设计标准（如 Flower, FATE, FedML）及架构设计基本原则（SOLID），对当前 `core/server/base_server.py` 及相关核心设计进行代码异味（Code Smells）和架构缺陷的“毒舌”诊断。部分评价可能极其刺耳，但问题致命，需尽快重构以满足学术科研的性能和扩展性需求。

---

## 1. 内存爆炸的定时炸弹 (OOM Time Bomb)

**❌ 致命表现：**
在 `BaseServer.broadcast` 中，你写下了这样一段代码：
```python
global_weights = {k: v.cpu().clone() for k, v in self.global_model.state_dict().items()}
package = {client_id: global_weights for client_id in selected_clients}
```

**⚠️ 毒舌诊断：**
这是极其典型的**新手实习生级代码**。如果你要仿真 1000 个客户端参与的训练，如果你的模型是 `ResNet-50` (约 100MB)，这段代码会**瞬间吃掉 100GB 的内存**，直接喜提 OOM Crash。如果换成目前科研里哪怕很小的 LLM（如 1GB Params），只要挑 10 个客户端你的机子就炸了。

**🔍 业界对比与改进：**
- **真实场景**：模型不应该在内存中被深度拷贝多份，应该转化为统一且只读的序列化字节流 (Byte Stream) 或使用 RPC 给各端发送指针/引用。
- **仿真场景 (Ray / 单机多进程)**：只产生**唯一拷贝**，存在 shared memory（或通过 Ray `put` 拿到 ObjectRef）中，所有被选中的 Client 只接收这个引用/指针。
- **建议**：立刻抛弃针对每个 client_id 构建独立深拷贝 Package 的做法。

## 2. 僵化的流水线设计 (Rigid Synchronous Pipeline)

**❌ 致命表现：**
在 `step()` 方法中，你硬编码了一条从上到下的防线：
`Screen -> Aggregate -> Base+Delta -> PostProcess`

**⚠️ 毒舌诊断：**
这是严重的**开闭原则 (OCP) 违背**。这种写法直接把 SafeFL 限死在了**最原始的同步联邦学习 (Sync FL)** 的耻辱柱上。
- 如果研究员想做**异步联邦 (FedAsync)** 怎么办？
- 如果有人想搞**去中心化联邦 (Decentralized FL / Gossip)** 或**多层级聚类聚合 (Hierarchical FL)** 怎么办？
难道让他们来改写 `BaseServer` 吗？那这个框架的抽象意义何在？

**🔍 业界对比与改进：**
- **真实案例**：Flower 采用了极致的 `Strategy` 设计模式；MMPreTrain/MMEngine 等采用了 Hook/Callback 生命周期的做法。
- **建议**：不要在 `base_server` 里硬编码 `Screen-Aggregate-Merge`。应该将其重构为事件驱动 (Event-driven) 或钩子模式 (Hooks)。比如：`on_before_aggregate()`, `aggregate()`, `on_after_aggregate()`，或者将这四步封装成一个可插拔的 `AggregationStrategy`。

## 3. 基本类型偏执 (Primitive Obsession) 与幽灵状态

**❌ 致命表现：**
整个流水线通过一个裸露的 `context = {}` 对象传递：
```python
context = {}
screen_scores, context = self.screener.screen(..., context=context)
```

**⚠️ 毒舌诊断：**
这是《重构》中典型的**“基本类型偏执 (Primitive Obsession)”**和**“数据泥团 (Data Clumps)”**。
你以为用个 `dict` 是“灵活”，实际上对下游的研究用户来说这是一场**“抓瞎式编程”**。我作为一个新防御算法的研究员，我怎么知道上一级的 `Screener` 在这个破字典里面塞了什么 key？一旦拼写错误，排查到吐血。

**🔍 业界对比与改进：**
- **建议**：杜绝裸奔的 `Dict`。引入 Python `dataclass` 建立强类型的 `ServerContext` 或 `ServerState`。明确定义它具有 `round_num`, `selected_clients_info`, `anomaly_scores` 等属性类型。保证 IDE 能够给出类型提示(Type Hints)。

## 4. 可笑并高度耦合的客户端选择逻辑

**❌ 致命表现：**
```python
def select_clients(self, client_ids: List[str], num_select: int) -> List[str]:
    return self.rng.sample(client_ids, num_select)
```

**⚠️ 毒舌诊断：**
将随机选择逻辑写死在 `BaseServer` 中，违背了 **单一职责原则 (SRP)**。这年头谁还在学术界发单纯只用 Random Client Selection 的顶会 Paper？大家都在做基于系统异构性 (如 Oort 选快的)、数据异构性 (Active FL, 选 Loss 大的) 的智能选择框架。

**🔍 建议与改进：**
- 将 `Selector` 抽离。将初始化参数中加上 `client_selector: BaseSelector`。
- `BaseServer` 只负责调用 `client_selector.select(...)` 拿到名单。把所有状态（如：历史延迟、统计梯度等）交给 `ClientSelector` 维护！

---

## 5. 极其业余且硬编码的梯度运算

**❌ 致命表现：**
在 `compute_delta` 中出现了这样的代码：
```python
if "num_batches_tracked" in key:
    continue
```

**⚠️ 毒舌诊断：**
这是极其刺眼的**坏味道 (Bad Smell)**！
1. **违背开闭原则**：凭什么写死过滤 `num_batches_tracked`？如果后续用大语言模型引入了 LayerNorm？如果引入了特殊的 Lora Rank 参数不需要同步？是不是又要来改 `BaseServer`？
2. **性能灾难**：你把所有 client_state_dict 全部集中在 Server 上做个巨型的 `for` 循环相减来算 Delta？在真实场景中，带宽已经被干爆了。如果非要服务端算，至少也应该是并行的。

**🔍 建议与改进：**
- **架构重构**：Delta 的计算应该在 **Client 端** 完成！Client 训练完后直接在本地计算 `w_new - w_old`，然后向 Server 上传 Delta。既省带宽又能分散计算压力。
- **过滤器抽象**：如果必须由服务器进行特殊逻辑过滤，抽象出一个 `ParameterFilter` 或在使用 State Dict 前通过一个清晰的 `ignore_keys` 列表过滤，绝对不允许在 Base 类出现写死的具体业务字符串。

---

### 总结与改造优先级

| 致命程度 | 问题描述 | 建议动作 |
|---------|----------|----------|
| 💥 紧急 | `broadcast` 直接导致深拷贝内存泄漏 | 改为只读全局引用分配，或构建统一的不可变对象。 |
| 💣 高危 | 业务硬编码 (`num_batches_tracked` 过滤) | 引入白名单/黑名单列表，把计算重包分放给 Client。 |
| 🚧 中高 | Rigid Pipeline | 引入 Strategy/Callback/Hooks 来控制 Server 执行流，取代直接的硬编码调用。 |
| 🚧 中等 | `context` 全局隐式字典 | 改为强类型 `dataclasses` (如 `ServerState`)。 |
| 🪚 中低 | `select_clients` 耦合 | 抽取 `BaseSelector`，支持可拔插的评估策略。 |
