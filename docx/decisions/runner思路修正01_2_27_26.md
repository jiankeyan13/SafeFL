# Runner（胶水层）重构思路修正

> **背景**: 当前 `FederatedRunner` (~476行) 承担了过多职责，与新版 `BaseClient`（极简化 `step()` 模板方法）和 `Evaluator` 的设计理念产生了严重的冲突与冗余。本文档系统性地梳理 Runner 中每一块逻辑的去留，目标是让 Runner 回归**纯粹的胶水层**身份：只编排流程，不含业务逻辑。

## 二、应当删除的部分（及理由）

### 2.1 ❌ `_build_client_dataloader` (L170-188) — **必须删除**

**问题**: 创建一个临时 Client 对象，仅仅为了调用它的 `data_load` 方法构建 DataLoader，然后立刻 `del`。这是典型的**滥用对象 (Object Abuse)** 反模式。

**矛盾**: 新版 `BaseClient` 的构造函数已经接收 `task_set` 和 `stores`，并在内部通过 `_build_dataset` + `_create_train_dataloader` 自行完成数据构建。Runner 不应自己构建 DataLoader。

**结论**: 删除。Server 端评估用的 DataLoader 应该由 `Evaluator` 或一个简单的工厂函数直接构建，不需要经过 Client。

---

### 2.2 ❌ `_eval_model` (L426-457) — **必须删除**

**问题**: 这是一个完整的评估循环 (`model.eval()` → `torch.no_grad()` → forward → 汇总 preds/targets → 计算 metric)。这块逻辑与已有的 `Evaluator.evaluate()` **100% 功能重复**。

**更深层的问题**: Runner 直接触碰了 `self.server.global_model`，破坏了 Server 的封装性。一个胶水层不应该知道模型推理的细节。

**结论**: 删除。全局评估统一调用 `Evaluator.evaluate(model, loader, criterion, device)`。

---

### 2.3 ❌ `_build_metrics` (L380-389) — **必须删除**

**问题**: 通过 `METRIC_REGISTRY.build()` 动态构建 Metric 对象列表。但新版 `Evaluator` 在初始化时接受 `metrics: Dict[str, BaseMetric]`，Metric 对象的构建天然是 `Evaluator` 初始化的一部分。

**结论**: 删除。在 setup 阶段构造 `Evaluator` 时一次性传入 metrics 配置即可。

---

### 2.4 ❌ `_eval_clean_global` + `_eval_attack_groups` (L275-320) — **必须删除/大幅重构**

**问题**:
1. `_eval_clean_global` 内部调用 `_eval_model`、`_build_metrics`，两者都将被删除。
2. `_eval_attack_groups` 中硬编码了"遍历攻击组 → 构建攻击策略 → 构建毒化 DataLoader → 评估"的完整流水线。这里存在：
   - **Runner 越权**: Runner 不应该自己去 `ATTACK_REGISTRY.build()`，攻击策略的构建是 `AttackManager` 的职责。
   - **重复代码**: 再次调用 `_build_client_dataloader`（创建临时 Client）。
   - **违背 SRP**: 攻击评估的编排逻辑不属于胶水层。

**结论**: 删除这两个方法。重构为统一的 `_evaluate(round_idx)` 方法，内部调用 `Evaluator`，只做编排。

---

### 2.5 ❌ `_setup_server_proxy_loader` (L137-149) — **应当删除**

**问题**: 这个方法唯一的作用是为 Server `step()` 中的 BN 校准构建一个 DataLoader。但：
1. 它再次调用了 `_build_client_dataloader`（临时 Client 反模式）。
2. BN 校准所需的 proxy DataLoader 应该在 Server 初始化或 Algorithm Builder 构建阶段就准备好，而非由 Runner 手动拼装再传入。

**结论**: 删除。将 proxy loader 的构建移入 Algorithm Builder 或 Server 构造过程中。

---

## 三、应当保留的部分

### 3.1 ✅ `__init__` 初始化骨架

保留：config 解析、device 判定、logger 构建、seed 设置、`_setup()` 调用链。
这是 Runner 的核心职责，无可替代。

### 3.2 ✅ `_set_seed` (L68-78)

全局种子设置是实验可复现性的根基，属于 Runner（实验管控者）的天然职责。保留。

### 3.3 ✅ `_setup_model` (L103-108)

`model_fn` 构建 + 全局模型初始化，简洁精准，保留。

### 3.4 ✅ `_setup_attack_manager` (L151-160)

攻击角色分配是实验场景设定的一部分，属于 Runner 职责。保留。

### 3.5 ✅ `run` 主循环骨架 (L190-235)

这是 Runner 存在的最核心理由——驱动 Round 循环。骨架保留，但内部的评估调用点需要简化。

### 3.6 ✅ `_save_checkpoint` (L369-378)

Checkpoint 管理是实验管控职责，保留。

### 3.7 ✅ `_aggregate_client_metrics` (L459-476)

训练指标的加权平均聚合属于 Runner 的汇总报告职责，保留。但可以考虑简化（例如内联到 `run` 循环中，因为逻辑只有 5 行）。

## 四、应当精简/重构的部分

### 4.1 ⚠️ `_setup_data_pipeline` (L80-101) — **抽离为独立模块**

**问题**: Runner 手动 if-else 选择 Partitioner，然后顺序构建 TaskGenerator、TaskSet、DatasetStore。这段代码虽然不复杂，但：
1. **Partitioner 的选择逻辑**不属于 Runner。如果后续增加新的 Partitioner（Pathological, Shard-based），需要改 Runner。
2. 整段可以封装为一个纯函数 `build_data_pipeline(config) → (TaskSet, DatasetStores)`。

**建议**: 将 Partitioner 也加入 Registry，通过 `PARTITIONER_REGISTRY.build()` 消除 if-else。整段逻辑抽为 `data/pipeline.py` 中的工厂函数。Runner 只一行调用即可：
```python
self.task_set, self.dataset_stores = build_data_pipeline(self.config['data'], self.seed)
```

---

### 4.2 ⚠️ `_setup_server_and_algorithm` (L110-136) — **精简**

**问题**:
1. L117-119 手动构建 server 端 test DataLoader（`Subset` + `DataLoader`），这不是 Runner 的活。
2. `ALGORITHM_REGISTRY.build()` 的调用本身是合理的，但返回值 `(server, client_class)` 强耦合了 Algorithm Builder 的接口设计。

**建议**:
- 删掉手动构建 test DataLoader 的代码，让它随 Evaluator 初始化或 Algorithm Builder 一起创建。
- `lr_scheduler` 的构建保留在此处。

---

### 4.3 ⚠️ `_run_local_training` (L237-264) — **与新 BaseClient 对齐**

**当前问题**: 该方法使用老版 Client API：
```python
client = self.client_class(cid, self.device, self.model_fn)
payload = client.execute(global_state_dict=..., task=..., dataset_store=..., ...)
```

**新版 BaseClient API**:
```python
client = BaseClient(client_id, task_set, stores, model, config, evaluator)
payload = client.step(server_payload)
```

**建议**: 重写此方法以对齐新 API。新版 Client 在构造时就拥有 task_set 和 stores 的引用，不需要 Runner 传递。逻辑会变得更简洁：
```python
def _run_local_training(self, selected_ids, server_payloads, round_config):
    updates = []
    for cid in selected_ids:
        client = self._create_client(cid, round_config)
        payload = client.step(server_payloads[cid])
        updates.append(payload)
    return updates
```

---

### 4.4 ⚠️ `_select_clients` (L391-422) — **逻辑正确但需精简**

**问题**: 攻击者强制采样 vs 随机采样的分支逻辑过于冗长（30行），但功能逻辑本身是对的——这确实是 Runner 编排"谁参与本轮"的核心职责。

**建议**: 保留，但将 attacker 采样逻辑委托给 `AttackManager`，让 Runner 只关心最终的 selected_ids 列表。代码应缩减到 ~10 行。

---

### 4.5 ⚠️ `_run_local_evaluation` (L322-367) — **精简**

同 `_run_local_training`，需要对齐新版 Client API。`client.evaluate()` 在新版 BaseClient 中已是一个独立方法，内部结合 Evaluator 自行完成。Runner 只需要：
1. 采样一批客户端。
2. 调用 `client.evaluate()` 收集结果。
3. 聚合并 log。

---

## 五、应当新引入的部分

### 5.1 🆕 统一的 `Evaluator` 集成

**核心思想**: Runner 不应自己写评估循环。应在 `_setup` 阶段构造一个 `Evaluator` 实例（或多个——clean 评估用一个，各攻击组评估各一个），在 `run()` 循环中直接调用。

```python
# setup 阶段
self.evaluator = Evaluator(metrics={"accuracy": Accuracy(), "loss": AverageLoss()})

# run 循环中
clean_results = self.evaluator.evaluate(self.server.global_model, self.test_loader, criterion, self.device)
```

这样 Runner 的全局评估代码从 ~80 行缩减到 ~5 行。

---

### 5.2 🆕 `_create_client` 工厂方法

将 Client 的创建逻辑统一封装，避免在 `_run_local_training` 和 `_run_local_evaluation` 中重复写构造代码。同时此方法是攻击注入的唯一入口：

```python
def _create_client(self, cid: str, round_config: Dict) -> BaseClient:
    """构造单个客户端实例，含攻击策略注入。"""
    attack_profile = self.attack_manager.get_strategy(cid) if self.attack_manager else None
    return self.client_class(
        client_id=cid,
        task_set=self.task_set,
        stores=self.dataset_stores,
        model=self.model_fn(),
        config=round_config,
        evaluator=self.evaluator,
        attack_profile=attack_profile,
    )
```

---

### 5.3 🆕 数据管道的工厂外移

将 `_setup_data_pipeline` 中的逻辑抽离为 `data/pipeline.py::build_data_pipeline()`，让 Runner 只调用一行。这同时为未来单元测试 Runner 提供了可 mock 的注入点。

---

### 5.4 🆕 `_build_eval_loaders` — 评估数据加载器的统一构建

将 Server 端 test DataLoader 和各攻击组的 poisoned DataLoader 的构建统一到一个方法中，在 setup 阶段一次性完成（因为这些 DataLoader 全程不变）：

```python
def _build_eval_loaders(self):
    """一次性构建所有评估用 DataLoader (clean + per-attack-group poisoned)。"""
    # clean
    self.eval_loaders = {"clean": build_loader(self.server_test_task, self.server_dataset_store)}
    # poisoned per group
    for group_name, group_config in attack_configs.items():
        poisoned_ds = PoisonedDataset(base_ds, strategy)
        self.eval_loaders[group_name] = DataLoader(poisoned_ds, ...)
```

这让 `run()` 循环中的评估逻辑彻底变成纯粹的遍历 + 调用 Evaluator。

## 六、重构后的 Runner 目标结构

```
FederatedRunner (~200行)
│
├── __init__(config)              # 解析 + _setup
├── _set_seed(seed)               # 种子
├── _setup()                      # 编排初始化
│   ├── task_set, stores ← build_data_pipeline(config)    # [外部]
│   ├── model_fn, global_model ← _setup_model()            # [内部]
│   ├── server, client_class ← ALGORITHM_REGISTRY.build()  # [内部]
│   ├── attack_manager ← AttackManager(config, ids, seed)  # [内部]
│   ├── evaluator ← Evaluator(metrics)                     # [新增]
│   └── eval_loaders ← _build_eval_loaders()               # [新增]
│
├── run()                         # 主循环 (~30行)
│   ├── for round in rounds:
│   │   ├── _select_clients()
│   │   ├── server.broadcast()
│   │   ├── _run_local_training()
│   │   ├── server.step()
│   │   ├── _evaluate(round_idx)   # 统一评估入口
│   │   └── _save_checkpoint()
│   └── logger.close()
│
├── _create_client(cid, config)   # [新增] 客户端工厂
├── _run_local_training(...)      # 遍历 client.step()
├── _select_clients(round_idx)    # 选人（含攻击者强制采样）
├── _evaluate(round_idx)          # [新增] 遍历 eval_loaders + evaluator
├── _save_checkpoint(round_idx)   # 保存
└── _aggregate_client_metrics()   # 训练指标汇总
```

**目标行数**: ≤ 200 行（含注释），相比当前 476 行削减 **~58%**。

## 七、核心设计原则总结

| 原则 | 当前违背 | 重构后 |
|------|----------|--------|
| **SRP (单一职责)** | Runner 既编排流程、又构建 DataLoader、又跑推理循环 | Runner 只编排，推理委托 Evaluator，数据构建委托工厂 |
| **DRY (不重复)** | `_eval_model` 与 `Evaluator.evaluate` 100% 重复 | 统一由 Evaluator 完成 |
| **OCP (开闭原则)** | 新增评估指标/攻击组需改 Runner | 指标由 Evaluator 管理，攻击组通过 config 驱动 |
| **LoD (迪米特法则)** | Runner 直接操作 `self.server.global_model` | Runner 只通过 Server/Evaluator 的公共接口交互 |
| **对齐新 Client** | 使用旧版 `client.execute(task=..., store=...)` API | 使用新版 `client.step(server_payload)` |

---

> **行动优先级**: 删 `_eval_model` + `_build_metrics` → 集成 `Evaluator` → 对齐新 `BaseClient` API → 外移数据管道 → 精简 `_select_clients`
