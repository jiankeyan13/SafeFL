# BaseClient 实现思考与设计决策

> **日期**: 2026-02-28  
> **上下文**: 数据层 (`data/`) 已完成重构，Server 端大体逻辑已就绪。本文档聚焦 `BaseClient` 的完整实现，确保它与数据层 (`TaskSet` + `DatasetStore`) 和胶水层 (`Runner`) 的对接干净利落。

## 一、当前 BaseClient 骨架审查

当前 `base_client.py` 定义了 7 个方法（含 `__init__`），全部是空的 `pass`。签名设计的方向是正确的——极简、直白、无回调。但有几个关键问题需要在实现时明确。

### 1.1 构造函数签名 — 需要修正

**当前签名**:
```python
def __init__(self, client_id: int, task_set: TaskSet, stores: Dict[str, DatasetStore], 
             model: Optional[nn.Module] = None, config: Optional[Dict[str, Any]] = None, 
             evaluator: Optional[Evaluator] = None)
```

**问题**:

| 问题 | 分析 |
|------|------|
| `client_id: int` | `TaskSet` 内部用的是 `str` 类型的 `owner_id`（如 `"client_0"`），但这里传 `int`。需要内部转换 `client_owner(client_id)` 才能从 `TaskSet` 查找。类型不一致会制造困惑。**建议**: 同时接受 `int` 和 `str`，内部统一为 `owner_id: str`。这样 Runner 侧传 `str` 更直接（因为 `task_set.list_client_ids()` 返回的就是 `str`）。 |
| `model: Optional` | 模型不应该是 Optional。没模型的 Client 毫无意义。**建议**: 改为 `model: nn.Module`，必传。 |
| `config: Optional` | 训练超参（lr、epochs、batch_size）是训练必需的。**建议**: 改为 `config: Dict[str, Any]`，必传。 |
| `evaluator: Optional` | 合理。没有则用默认内置的 `Evaluator()`，有则用外部传入的。保持不变。 |
| 缺少 `device` | 模型在哪个设备上跑？当前签名里没有 `device`。模型本身可能在 CPU 上构建，需要 `.to(device)` 后才能训练。**建议**: 增加 `device: torch.device` 参数。 |

**修正后的签名**:
```python
def __init__(
    self,
    client_id,                              # int 或 str 均可
    task_set: TaskSet,
    stores: Dict[str, DatasetStore],
    model: nn.Module,                        # 必传
    device: torch.device,                    # 新增
    config: Dict[str, Any],                  # 必传
    evaluator: Optional[Evaluator] = None,
)
```

### 1.2 `__init__` 内部实现

```python
def __init__(self, client_id, task_set, stores, model, device, config, evaluator=None):
    # client_id 兼容 int/str
    if isinstance(client_id, int):
        self.client_id = client_id
        self.owner_id = client_owner(client_id)
    else:
        self.owner_id = client_id             # 已是 "client_0" 形式
        self.client_id = int(client_id.split('_')[-1])
    
    self.task_set = task_set
    self.stores = stores
    self.model = model.to(device)
    self.device = device
    self.config = config
    self.evaluator = evaluator or Evaluator()
    
    # 构造时主动构建 DataLoader（直白，无懒加载）
    self.train_loader = self._create_train_dataloader()
    self.val_loader = self._create_val_dataloader()
```

**关于"无状态"的原则**: Client 实例是临时的——Runner 每轮 `_create_client()` 创建、调用 `step()`、收集 payload、然后 `del`。所以持有 `self.model` 不会导致 OOM（因为同一时刻只有一个 Client 在运行，运行完就销毁）。这和之前"在 `__init__` 里给 100 个 Client 都分配模型"的灾难完全不同。

## 二、`_build_dataset` 实现 — 数据对接核心

### 数据流解析

```
TaskSet                              DatasetStore (stores dict)
  └─ try_get_task(owner_id, split)     └─ stores[task.dataset_tag]
       → Task(owner_id, dataset_tag,       → DatasetStore(name, split, dataset)
              split, indices)                    ↓
                                           Subset(store.dataset, task.indices)
                                               ↓
                                           DatasetStore(name, split, Subset(...))
```

**关键认知**: `Task.indices` 是针对 `stores[task.dataset_tag].dataset` 的索引。构建流程：
1. 从 `task_set` 拿到 `Task` 对象（含 `dataset_tag` 和 `indices`）。
2. 从 `stores` 字典中用 `dataset_tag` 拿到对应的 `DatasetStore`。
3. 用 `Subset(store.dataset, task.indices)` 切出子集。
4. 用新的 `DatasetStore` 包装，使其仍然拥有 `get_label()` 等方法。

**实现**:
```python
def _build_dataset(self, split: str) -> Optional[DatasetStore]:
    task = self.task_set.try_get_task(self.owner_id, split)
    if task is None:
        return None
    store = self.stores[task.dataset_tag]
    subset = Subset(store.dataset, task.indices)
    return DatasetStore(
        name=f"{self.owner_id}_{split}",
        split=split,
        dataset=subset,
    )
```

> **注意**: `train` split 的 `dataset_tag` 指向带增强的 store (`cifar10_train_aug`)，而 `test` split 的 `dataset_tag` 指向无增强的 store (`cifar10_train_plain`)。`TaskGenerator._build_client_tasks()` 已经做好了映射，`_build_dataset` 不需要关心增强/非增强的区别。

## 三、DataLoader 构建

### 3.1 `_create_train_dataloader`

```python
def _create_train_dataloader(self) -> DataLoader:
    ds = self._build_dataset(SPLIT_TRAIN)
    if ds is None:
        raise RuntimeError(f"Client {self.owner_id} has no training data.")
    batch_size = self.config.get('batch_size', 64)
    return DataLoader(ds, batch_size=batch_size, shuffle=True, drop_last=False)
```

**子类扩展点**: 攻击者可以重写此方法，在 `ds` 上先调用 `self.attack_strategy.poison_dataset(ds)` 再构建 DataLoader。

## 四、`receive` — 接收服务端下发

`BaseServer.broadcast()` 返回的是 `Dict[str, Dict[str, Tensor]]`，外层 key 是 client_id，内层是纯 state_dict。Runner 选出对应 Client 的 state_dict 传入 `receive()`。

```python
def receive(self, server_payload: Dict[str, Any]) -> None:
    """server_payload: 来自 server.broadcast() 的 state_dict"""
    self.model.load_state_dict(server_payload, strict=False)
```

简洁明了。如果未来需要传递更多信息（如 learning rate、round_idx），扩展 `BaseServer.broadcast()` 即可。

## 五、`train` — 本地训练

最核心的方法。**裸奔的原生 PyTorch 循环**——研究者追求的就是这种透明度。

```python
def train(self) -> Dict[str, Any]:
    self.model.train()

    optimizer = torch.optim.SGD(
        self.model.parameters(),
        lr=self.config.get('lr', 0.01),
        momentum=self.config.get('momentum', 0.9),
        weight_decay=self.config.get('weight_decay', 1e-4),
    )
    criterion = nn.CrossEntropyLoss()
    local_epochs = self.config.get('local_epochs', 5)
    
    total_loss = 0.0
    total_samples = 0
    
    for epoch in range(local_epochs):
        for data, target in self.train_loader:
            data, target = data.to(self.device), target.to(self.device)
            
            optimizer.zero_grad()
            output = self.model(data)
            loss = criterion(output, target)
            loss.backward()
            optimizer.step()
            
            total_loss += loss.item() * target.size(0)
            total_samples += target.size(0)
    
    avg_loss = total_loss / total_samples if total_samples > 0 else 0.0
    return {"train_loss": avg_loss}
```

**设计决策**:

| 决策 | 理由 |
|------|------|
| Optimizer 在 `train()` 内创建 | Client 是临时对象，`train()` 只调用一次。在内部创建更内聚。攻击者重写 `train()` 可自由换用 Adam 等。 |
| 不抽象 criterion / optimizer | 99% 联邦后门攻防用 `SGD` + `CrossEntropyLoss`。个别算法需求直接重写 `train()` 即可。|
| 不使用 Callback | 研究员直接看到完整的训练循环，白盒透明。需要改什么就覆写什么。 |

## 六、`evaluate` — 本地评估

```python
def evaluate(self) -> Dict[str, float]:
    if self.val_loader is None:
        return {}
    criterion = nn.CrossEntropyLoss()
    return self.evaluator.evaluate(self.model, self.val_loader, criterion, self.device)
```

委托给 `Evaluator`，不在 Client 里重复写评估循环。

## 七、`package` — 打包上传

```python
def package(self) -> Dict[str, Any]:
    return {
        "client_id": self.owner_id,
        "weights": {k: v.cpu().clone() for k, v in self.model.state_dict().items()},
        "num_samples": len(self.train_loader.dataset),
    }
```

> **重要**: 返回 `"weights"` 键名与 `BaseServer.step()` 中读取 `up['weights']` 对齐。  
> **攻击者扩展**: `package()` 是模型投毒 (Scaling Attack) 的关键覆写点。子类可先 `payload = super().package()`，然后对 `payload['weights']` 做手脚。

## 八、`step` — 模板方法

```python
def step(self, server_payload: Dict[str, Any]) -> Dict[str, Any]:
    self.receive(server_payload)
    train_metrics = self.train()
    eval_metrics = self.evaluate()
    payload = self.package()
    payload["metrics"] = train_metrics
    if eval_metrics:
        payload["eval_metrics"] = eval_metrics
    return payload
```

## 九、与胶水层 (Runner) 的对接协议

### 9.1 Runner 需要给 Client 什么

| 参数 | 来源 | 说明 |
|------|------|------|
| `client_id` | `selected_ids` 列表中的 `str` | `task_set.list_client_ids()` 返回 `str`（如 `"client_0"`） |
| `task_set` | `self.task_set` | Runner 持有，直接传入 |
| `stores` | `self.dataset_stores` | Runner 持有，直接传入 |
| `model` | `self.model_fn()` | **每次创建新实例**，避免串扰 |
| `device` | `self.device` | Runner 持有 |
| `config` | `round_config` | 含当前轮 lr、batch_size 等 |
| `evaluator` | `self.evaluator` | Runner 构造时创建的共享实例 |

### 9.2 Runner `_create_client` 工厂方法

```python
def _create_client(self, cid: str, round_config: Dict) -> BaseClient:
    return self.client_class(
        client_id=cid,
        task_set=self.task_set,
        stores=self.dataset_stores,
        model=self.model_fn(),
        device=self.device,
        config=round_config,
        evaluator=self.evaluator,
    )
```

### 9.3 Runner 主循环调用

```python
def _run_local_training(self, selected_ids, server_payloads, round_config):
    updates = []
    for cid in selected_ids:
        client = self._create_client(cid, round_config)
        payload = client.step(server_payloads[cid])
        updates.append(payload)
        del client
    return updates
```

## 十、未来攻击者扩展路径 (仅供参考，暂不实现)

```python
class MaliciousClient(BaseClient):
    """攻击者客户端 —— 覆写数据加载以注入数据投毒"""

    def __init__(self, *args, attack_strategy: AttackStrategy, **kwargs):
        self.attack_strategy = attack_strategy
        super().__init__(*args, **kwargs)  # 会触发 _create_train_dataloader

    def _create_train_dataloader(self) -> DataLoader:
        ds = self._build_dataset(SPLIT_TRAIN)
        poisoned_ds = self.attack_strategy.poison_dataset(ds, mode='train')
        batch_size = self.config.get('batch_size', 64)
        return DataLoader(poisoned_ds, batch_size=batch_size, shuffle=True)

    def package(self) -> Dict[str, Any]:
        payload = super().package()
        # 模型投毒
        payload['weights'] = self.attack_strategy.poison_update(
            payload['weights'], self._initial_weights
        )
        return payload
```

> 攻击者只需覆写 2~3 个方法即可完成攻击注入，完全不需要回调系统或 Wrapper 模式。

## 十一、总结

| 设计要点 | 决策 |
|----------|------|
| Client 是否持有模型 | ✅ 是，但 Client 是临时对象，用完即销毁 |
| Optimizer 在哪里创建 | `train()` 内部，更内聚 |
| 数据构建逻辑 | `_build_dataset` → `_create_*_dataloader`，两级解耦 |
| 评估逻辑 | 完全委托给 `Evaluator` |
| 攻击者如何扩展 | 继承 + 覆写，不需要任何额外抽象 |
| `client_id` 类型 | 同时接受 `int`/`str`，内部统一为 `owner_id: str` |
| 与 Runner 的契约 | `step(server_payload) → payload_dict` |
